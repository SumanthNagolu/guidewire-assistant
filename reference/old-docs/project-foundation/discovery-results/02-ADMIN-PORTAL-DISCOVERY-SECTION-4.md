## **üéØ CRITICAL ARCHITECTURAL DECISION: GUIDEWIRE-STYLE ACTIVITY SYSTEM**

**Core Concept:**
- **Activity-Driven Workflow:** Every work item (job, candidate, placement, lead) is broken down into sequential activities
- **Activity Table:** Central table linked to ALL core entities
- **Ownership Model:** Two owners per activity
  - **Responsible:** Person doing the work
  - **Accountable:** Person overseeing/approving

**Activity Types by Department:**

### **1. RECRUITING (Role-Specific Activities)**
**Job lifecycle activities:**
1. Receive job requirement
2. Polish/refine job description
3. Post on job board
4. Source candidates
5. Screen candidates
6. Submit to client
7. Schedule interview
8. Follow up post-interview
9. Offer negotiation
10. Placement

**Characteristics:** Sequential, role-specific, time-bound

### **2. BENCH SALES (Recurring/Extended Activities)**
**Candidate management activities:**
- Add to bench
- Market to clients (RECURRING - ongoing until placed)
- Client interested
- Interview/screening
- Negotiation
- Placement
- Ongoing management

**Characteristics:** Recurring, can be extended, same activity continues for long time

### **3. TALENT ACQUISITION (Lead-Specific Activities)**
**Lead management activities:**
- Identify lead
- Initial contact
- Qualification
- Pitch/presentation
- Negotiation
- Onboard lead/client

**Characteristics:** Target-based, lead-specific, conversion-focused

### **4. TRAINING (Day-Specific Activities)**
**Daily/recurring activities:**
- Student progress check
- Assignment review
- Quiz grading
- Project evaluation
- Certification approval

**Characteristics:** Day-specific, can be split daily

**Activity System Architecture:**

```typescript
// Activity Table (Guidewire-style)
activity {
  id: string
  entity_type: 'job' | 'candidate' | 'lead' | 'placement' | 'student'
  entity_id: string  // FK to linked entity
  activity_type: string  // Based on workflow
  status: 'pending' | 'in_progress' | 'completed' | 'blocked'
  owner_responsible: user_id  // Person doing work
  owner_accountable: user_id  // Person overseeing
  due_date: timestamp
  priority: 'high' | 'medium' | 'low'
  is_recurring: boolean  // For bench sales, ongoing activities
  recurrence_rule: string  // If recurring
  completion_date: timestamp
  notes: text
  attachments: []
  created_at: timestamp
  updated_at: timestamp
}
```

**Key Features:**
- [x] Activity table linked to all core entities (flexible FK system)
- [x] Sequential activities (workflow orchestration)
- [x] Recurring activities (bench sales, ongoing work)
- [x] Two-owner system (responsible + accountable)
- [x] Activity-based tracking ("If it's not tracked as activity, it's not done")
- [x] Can break down activities into sub-activities if needed
- [x] Activity completion = workflow progress

**This is the CORE of how work gets tracked and managed across all departments.**
